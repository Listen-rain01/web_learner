# 代码生成策略

本文档定义本项目对代码生成的使用边界。

## 总原则

代码生成是辅助手段，不是架构核心。

本项目的主要复杂度来自：
- 非标准协议
- 自定义编码
- HTML 解析
- 多步骤流程编排

因此，不允许为了少写样板而把核心协议层交给生成器。

## 当前允许使用的生成方案

允许考虑：
- `json_serializable`
- `freezed`

## 允许使用的位置

### `json_serializable`

适合：
- `features/*/data/models/`
- 纯 DTO
- 纯字段映射对象

不适合：
- 协议流程对象
- HTML 解析对象
- repository 实现

### `freezed`

适合：
- 复杂状态对象
- 明确需要不可变建模的 domain/entity
- 多状态联合类型

不适合：
- 每个简单类都强行生成
- 为了 `copyWith` 而过度引入复杂度

## 当前阶段不建议引入

当前阶段不建议：
- `retrofit`
- `riverpod_generator`
- `auto_route`
- `injectable`

原因：
- 会掩盖真实协议复杂度
- 让协议适配层更难调试
- 在项目尚未稳定前引入额外系统复杂度

## 明确禁止的目录

以下目录禁止依赖自动接口生成思路：
- `core/network/`
- `core/parsing/`
- `features/*/data/remote/`
- `features/*/data/repositories/`
- `app/router/`

这些位置必须以手写实现为主。

## 后续启用代码生成的条件

只有当某个模块满足以下条件时，才建议正式引入代码生成：
- 协议已经稳定
- 字段结构已经稳定
- 测试样例已经具备
- 生成前后不会削弱可调试性

## 结论

本项目的策略不是“禁止代码生成”，而是：

只在 DTO 和复杂状态对象中使用代码生成，  
协议层、解析层、仓储层、路由层坚持手写。
