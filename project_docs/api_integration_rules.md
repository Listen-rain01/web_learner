# 接口接入约束

本文档约束本项目如何接入真实接口，尤其适用于非标准 Web 协议。

## 总原则

接口接入优先考虑：
- 正确性
- 可调试性
- 可测试性

不要优先追求：
- 少写几行样板代码
- 形式上的自动生成

## 客户端约束

统一要求：
- HTTP 客户端统一使用 `Dio`
- 不允许页面层直接 new 客户端
- 会话统一通过 Cookie Session 管理

## 请求编码

对于自定义编码逻辑，例如 `Esdt()`：
- 只能放在 `core` 的编码能力中，或在 feature 的 remote 层通过统一编码器调用
- 页面层不能直接拼接编码
- repository 层不应手写重复编码细节

## HTML 与页面载荷

允许解析 HTML 的位置只有两类：
- `core/parsing`
- `features/<feature>/data/remote`

禁止：
- 页面层直接解析 HTML
- repository 层直接堆大段 HTML 提取逻辑

## 远程数据源与仓储分工

`remote data source`
- 负责发请求
- 负责拿原始响应
- 负责最靠近协议层的适配

`repository`
- 负责多接口编排
- 负责把原始结果组装成业务语义
- 负责向 application 层暴露稳定方法

## 错误分层

至少区分以下错误：
- 网络错误
- 协议错误
- 解析错误
- 业务错误

禁止把所有失败都统一变成一个模糊字符串。

## 日志规则

允许记录：
- 接口路径
- 请求阶段
- 状态转换
- 非敏感上下文

禁止明文记录：
- 身份证号
- 密码
- Cookie
- 完整敏感响应体

## 真实接口接入前的准备

接一个新接口前，必须先明确：
- 它属于哪个 feature
- 它的鉴权方式
- 它是否依赖上一步流程上下文
- 它的请求体编码方式
- 它的响应是 JSON、HTML 还是混合结构

## 测试要求

真实接口一旦接入，至少补其中一种：
- 编码器测试
- 解析器测试
- repository 流程测试
- widget 流程测试

复杂流程接口推荐使用 fixture 测试。
