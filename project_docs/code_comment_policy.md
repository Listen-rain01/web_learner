# 代码注释规范

本文档定义 `web_learner` 项目的代码注释约束。

本规范以 Dart 官方规范为主，适用于 Flutter / Dart 代码。

官方参考：

- Dart Language: Comments  
  <https://dart.dev/language/comments>
- Effective Dart: Documentation  
  <https://dart.dev/effective-dart/documentation>
- Effective Dart: Style  
  <https://dart.dev/effective-dart/style>

## 1. 总原则

注释的目标不是“让代码看起来更规范”，而是补充代码本身无法直接表达的信息。

必须优先遵守：

1. 代码优先自解释
2. 注释优先解释“为什么”
3. 公共 API 优先使用文档注释
4. 不写重复代码表面行为的废话注释

禁止把注释当作以下问题的替代品：

- 命名不清
- 结构过乱
- 函数过长
- 业务边界不明确

如果一段代码需要大量注释才能勉强看懂，应优先重构代码，而不是继续堆注释。

## 2. 注释类型

本项目只使用以下两类注释：

### 2.1 文档注释 `///`

用于：

- class
- enum
- mixin
- extension
- typedef
- 对外暴露的方法
- 对外暴露的重要属性

文档注释的目标是说明：

- 这个类型/方法是做什么的
- 它的业务职责
- 必要时补充使用限制、异常或上下文

示例：

```dart
/// 管理当前已登录用户的会话状态。
class AuthController extends Notifier<AuthState> {}
```

```dart
/// 使用账号和密码发起登录。
///
/// 当后端拒绝请求时抛出 [AppException]。
Future<void> signIn({
  required String account,
  required String password,
}) async {
  ...
}
```

### 2.2 实现注释 `//`

用于解释复杂实现细节，尤其是：

- 协议特殊性
- 后端接口坑点
- 解析规则
- 编码规则
- 顺序依赖
- 权限限制
- 临时兼容逻辑

示例：

```dart
// 后端要求身份证号在编码前先做归一化处理。
final normalizedAccount = _normalizeIdCard(account);
```

```dart
// MyBaseInfo 返回的是 HTML，不是 JSON，所以解析必须留在 remote 层。
final html = response.data as String;
```

## 3. 公共 API 规范

以下内容默认应写 `///` 文档注释：

- `core` 中对外复用的基础类
- `features/*/domain` 中的公开实体与仓储接口
- `features/*/application` 中的公开 controller / state / coordinator
- Provider 行为明显不直观时的说明

以下内容可以不写：

- 明显自解释的简单 getter / setter
- 无业务复杂度的简单数据类字段
- 仅内部使用的短小私有方法

要求：

1. 首句直接说明职责
2. 首句使用完整短句
3. 避免无信息量描述

不推荐：

```dart
/// 这个类用于处理登录。
class AuthController ...
```

推荐：

```dart
/// 处理认证状态与登录流程。
class AuthController ...
```

## 4. 实现注释使用规则

实现注释只在以下情况下允许出现：

1. 代码涉及真实接口文档中的特殊约定
2. 代码涉及后端返回格式陷阱
3. 代码涉及安全、权限、编码、脱敏规则
4. 代码中的顺序、条件或分支不能随意改动
5. 代码需要保留临时兼容逻辑

实现注释应该解释：

- 为什么这样做
- 不能改成什么
- 与哪条业务规则或接口规则对应

不推荐：

```dart
// 把 loading 设为 true。
state = state.copyWith(isLoading: true);
```

推荐：

```dart
// 登出完成前保留上一份会话，避免路由闪烁。
state = state.copyWith(isSubmitting: true);
```

## 5. 明确禁止的注释

以下注释在本项目中禁止出现：

### 5.1 翻译代码行为

例如：

```dart
// i 加一。
i++;
```

### 5.2 重复变量名含义

例如：

```dart
// userName 就是用户名。
final userName = ...
```

### 5.3 为了“显得规范”而写的无信息量注释

例如：

```dart
// 登录逻辑
// 按钮组件
// 构建界面
```

### 5.4 大段注释掩盖坏结构

如果注释必须长篇解释函数内部流程，优先拆函数。

## 6. 本项目特别需要注释的场景

本项目属于非标准 Web 协议对接项目，下列位置应积极补充高质量注释：

### 6.1 编码规则

例如：

- `Esdt()` 编码
- 明文参数和编码参数混用

### 6.2 HTML 载体接口

例如：

- `MyBaseInfo` 返回 HTML，不是 JSON
- 某些字段来自 HTML 内嵌 `vData`

### 6.3 多步骤业务流程

例如：

- 题库流程中单位、一级分类、子层节点、题库列表的依赖关系
- 某一步请求依赖上一阶段上下文

### 6.4 安全与脱敏规则

例如：

- 不保存明文密码
- 不保存完整身份证号
- Supabase 只做镜像资料，不做认证源

### 6.5 权限与准入规则

例如：

- 登录前准入检查
- `AllowChangeCurriculumVitae`
- 无权限时为什么隐藏/禁止某些动作

## 7. 与文档联动规则

如果代码实现明显依赖项目文档中的规则，应在必要处用注释点出对应事实。

例如：

- 个人信息接口文档说明 `MyBaseInfo` 承载基础资料
- 登录接口文档说明身份证号和密码都要先编码
- 题库接口文档说明子层节点不一定只是年份

注释不需要复制整段文档，只需简要指出关键事实。

## 8. 注释语言与风格要求

本项目代码注释统一使用中文。

统一要求：

1. `///` 文档注释使用简洁中文
2. `//` 实现注释使用简洁中文
3. 不允许中英混写，专有名词、类型名、接口名除外
4. 一条注释只表达一个重点
5. 注释内容应可直接落在具体代码附近
6. 优先短句，不写大段散文式说明

推荐：

```dart
// 这里返回的是 HTML，不是 JSON，所以解析必须留在 remote 层。
```

```dart
// 这里不能直接查 Supabase 公开表，登录准入必须走受控函数。
```

## 9. 最终执行要求

本项目最终执行以下规则：

1. 公共 API 优先写 `///`
2. 私有实现默认不写注释
3. 复杂协议、特殊业务规则用 `//` 解释“为什么”
4. 禁止无信息量注释
5. 注释不能替代重构
6. 代码注释统一使用中文

如果新增代码不满足以上约束，应优先修改代码结构或删除低质量注释。
