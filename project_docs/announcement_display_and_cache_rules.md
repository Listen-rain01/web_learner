# 登录页公告缓存与展示规则

本文档约束登录页公告的缓存、刷新与界面展示行为。

## 1. 基本原则

1. 公告真实数据源以 Supabase 为准。
2. 本地缓存只用于提升首屏展示速度，不作为最终真相。
3. 登录页公告属于辅助信息，不能阻塞登录主流程。

## 2. 有效公告定义

满足以下条件的公告才允许展示：

1. `id` 非空
2. `title` 非空
3. `content` 非空
4. 如果存在 `starts_at`，当前时间必须不早于开始时间
5. 如果存在 `ends_at`，当前时间必须不晚于结束时间

任一条件不满足，都视为“无可展示公告”。

## 3. 登录页展示规则

1. 登录页只在存在有效公告时显示公告区域。
2. 没有有效公告时，不显示公告卡片，也不显示“暂无公告”占位。
3. 本地缓存存在时，登录页优先展示缓存公告。
4. 首次安装且无缓存时，登录页允许不显示公告区域。

## 4. 缓存与刷新规则

1. 启动应用时先读取本地缓存。
2. 读取到缓存后，登录页应优先展示缓存内容。
3. 应用启动后立即在后台刷新一次最新公告。
4. 同一次应用运行期间，再次进入登录页时可按刷新间隔决定是否重复请求。
5. 本地缓存需要记录最近一次成功刷新时间。

## 5. 刷新结果处理规则

### 5.1 刷新成功且存在有效公告

1. 更新内存态公告
2. 覆盖本地缓存
3. 更新最近成功刷新时间

### 5.2 刷新成功但返回空列表

1. 立即清空内存态公告
2. 立即清空本地缓存
3. 更新最近成功刷新时间
4. 登录页隐藏公告区域

### 5.3 刷新成功但记录全部无效

处理方式与“返回空列表”一致。

### 5.4 刷新失败

1. 如果本地已有缓存，继续显示旧缓存
2. 不清空本地缓存
3. 不在登录页额外弹错误
4. 仅记录日志，供开发排查

## 6. 用户体验目标

1. 第二次及以后打开应用时，登录页公告应优先秒出缓存内容。
2. 如果后台已关闭公告，旧缓存只允许短暂存在，刷新成功后立即消失。
3. 登录页不长时间展示“正在加载公告”占位。
